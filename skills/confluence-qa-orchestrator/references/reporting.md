# Reporting Guidance

## Goal
Produce a useful report for stakeholders, not just a raw dump of results.

## Include
- project summary
- scope of requirements analyzed
- modules/features covered
- requirement coverage overview
- number of generated test cases
- execution summary
- pass/fail/blocked counts
- major defects or observations
- notable risks
- assumptions and open questions
- recommendations for next steps

## Formats
Generate:
- HTML report
- PDF report
- JSON run summary

## Report style
The report should be:
- concise
- structured
- readable by QA, PM, and engineering
- explicit about what was not tested

## Quality bar
The reader should quickly understand:
- what was in scope
- what was tested
- what happened
- what remains risky
